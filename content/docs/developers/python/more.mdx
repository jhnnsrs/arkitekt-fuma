So we just created our first app. And ran it against the Arkitekt Server. But what did we just do ? And saying my name is not really
useful. So lets dive a bit deeper and lets create a real app

## Adapting our real App

Lets adapt our app to do something useful. We will create a simple app that will allow us to max project a microscopy image of our
choice. So how do we do that ? Lets look at the code we created, before and deduce what we need to change.

```python
from arkitekt_next import register

@register
def hello_world(name: str) -> str:
    """Hello World

    This is a simple Hello World Action, that will greet you by your name.
    and print it to the console.

    Args:
        name (str): Your name

    Returns:
        str: A greeting
    """
    greeting = f"Hello {name}"
    print(greeting)
    return greeting
```

So what do we need to change? First of all we need to change the name of the function. We will call it `max_project`. Then we need to
change the documentation. We will call it `Max Project` and change the description to `Max Project a microscopy image`. Documenting
your functions is very important — it will help you and others to understand what your function does, and it will help users find your function
in the UI.

But how do we tell Arkitekt that we are no longer working with strings but need "Images"? This is done through updating the type hint of our function. We will change it from `str` to `Image`,
imported from the `mikro_next` Python package. When you register a function, the SDK will automatically detect the type hint and will create the correct input and output [Port](/docs/design/terminology/port) for you.
This means you have to (almost) always use type hints to indicate the type of your input and output.


```python
from arkitekt_next import register
from mikro_next.api.schema import Image, from_array_like

@register
def max_project(image: Image) -> Image:
    """Max Project a microscopy image

    This is a simple action that will max project a microscopy image along the z-axis.

    Args:
        image (Image): The input image

    Returns:
        Image: The max-projected output image
    """
    projected = image.data.max(dim="z")
    return from_array_like(projected, name=f"Max Projection of {image.name}", origins=[image])
```

And that is it.

<Callout type="info">


The `Image` type from `mikro_next` is our first [Structure](/docs/design/terminology/structure). Structures represent more complex data types than
simple types like `String` or `Int`. While most of the time you will find yourself working with sharable global structures (that live on a connected
service), you can also create your own structures (even non-serializable ones) that are only available to your app. We will cover this in a later tutorial.


</Callout>

## Advanced: App Lifecycle Hooks

`arkitekt-next` provides several decorators beyond `@register` to help you manage your app's lifecycle and share state between actions.

### `@startup` — Run code on agent start

Use `@startup` to run initialization logic once when your app connects to the server (e.g. loading a model into memory).

```python
from arkitekt_next import register, startup

@startup
async def load_model():
    # called once when the agent starts up
    print("Model loaded!")

@register
def run_inference(image: Image) -> Image:
    ...
```

### `@background` — Run persistent background tasks

Use `@background` to run an async function as a long-running background task while your app is connected.

```python
from arkitekt_next import register, background

@background
async def poll_hardware():
    import asyncio
    while True:
        # check hardware status, emit logs, etc.
        await asyncio.sleep(5)
```

### `@state` — Share mutable state between actions

Use `@state` to define a dataclass that holds shared state across all your actions. The state is initialized once and injected into any action that declares it as a parameter.

```python
from arkitekt_next import register, startup, state
from dataclasses import dataclass, field

@state
@dataclass
class MyState:
    model: object = field(default=None)

@startup
async def load(my_state: MyState):
    my_state.model = load_my_model()

@register
def run_inference(image: Image, my_state: MyState) -> Image:
    result = my_state.model.predict(image.data)
    return from_array_like(result, name="inference result")
```

### `@context` — Inject app-level context into actions

Use `@context` to define a context class that provides access to connected services (e.g. database clients) within your actions.

```python
from arkitekt_next import register, context

@context
class MyContext:
    ...  # populated automatically with service clients

@register
def my_action(image: Image, context: MyContext) -> Image:
    ...
```

These hooks give you full control over your app's lifecycle without any extra boilerplate.
