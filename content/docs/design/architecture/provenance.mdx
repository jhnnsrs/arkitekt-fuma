# Provenance & Audit Logging

The [Security](/docs/design/architecture/security) page makes a promise: that Arkitekt keeps a
sharp line between **you** and **the machine**, and that every change to your data is honestly
attributed. This page explains how that promise is kept.

The answer is **provenance-aware audit logging**. Every create, update, and delete on a tracked
record is captured as a history row, and each row is attributed to *who* made the change,
*which app* they used, and вҖ” crucially вҖ” the verified **task** it happened under. This lets an
administrator ask, after the fact, questions like:

> *"Which automated run touched this record, on whose authority, and what exactly did it change?"*

and get a precise, reliable answer.

<Callout type="info" title="Where this lives">
The provenance layer is implemented by a small Django library called **Koherent**, which the
Arkitekt services build on. If you are building or operating a backend service and want the
hands-on setup, see the [Koherent integration guide](/docs/developers/backend/koherent).
</Callout>

## What gets recorded

Every change produces a **provenance entry** that carries:

| Field               | Meaning                                                          |
|---------------------|------------------------------------------------------------------|
| `user`              | The human user the change is attributed to.                      |
| `client`            | The app / OAuth client that made the change, if any.             |
| `task`              | The verified task (assignation) the change ran under, if any.    |
| `kind`              | `CREATE`, `UPDATE`, or `DELETE`.                                 |
| `date`              | When the change happened.                                        |
| `effectiveChanges`  | The per-field old вҶ’ new diff for that change.                    |

This is what makes the human-versus-machine distinction concrete. When an agent runs a node on
your behalf, the entry records both facts at once: that the work was **authorized by you**, and
that it was **carried out by a machine** вҖ” never collapsing one into the other.

## What is a task (assignation)?

A **task** вҖ” also called an *assignation* вҖ” groups together every change made during one
logical run. In Arkitekt, when a user calls an app through [Rekuest](/docs/design/services/rekuest),
all of that app's changes carry the same task identity. So you can later find, audit, or revert
**every** change a single run produced, as one coherent unit.

Long-running tasks can span many separate requests over time; because a task is keyed by its
identity rather than by request, all of those changes still collapse onto the same task.

- **[kante](https://github.com/jhnnsrs/kante)** provides the GraphQL transport and the request
  context the upper layers build on.
- **[authentikate](https://github.com/jhnnsrs/authentikate)** owns identity. It verifies the
  signed provenance token and resolves the acting *user*, *client*, and *organization*. There is
  no bypass for provenance вҖ” the signature is always checked.
- **[koherent](https://github.com/jhnnsrs/koherent)** consumes that verified context and turns
  each data change into a durable, attributed history row, building a `Task` from the token's
  claims.

Because each layer depends on the one above it, the audit trail is only ever written from a
*verified* request вҖ” there is no path that records an unattributed or unsigned change.

## One token, many changes

The reason a task can group many changes is worth spelling out, because it is the heart of *why*
provenance is trustworthy вҖ” and it is not something the audit layer does by force. It falls out
of how a [rekuest-next](https://github.com/jhnnsrs/rekuest-next) agent runs a task.

When a user calls a registered function on an agent, the agent receives a single, server-signed
**provenance token** for that one run. While the function executes, every outgoing operation it
makes вҖ” every mutation, no matter how many вҖ” is transparently stamped with that **same** token.
The application code does nothing special; the token is set once and propagates to everything the
function does.

On the server side, the first change under that token creates the task, and every later change
with the same token reuses it. The result: every history row produced anywhere in that one run
links back to a single task, and you can later query *"show me everything this run changed."*

## Querying and reverting

Because provenance is recorded as first-class data, it is exposed directly in each service's
GraphQL API as a queryable, filterable `provenance` field on the records that carry it. You can
ask a record for its full history, filter records by the task or user that changed them, and вҖ”
for most actions вҖ” revert a change to restore the previous state.
