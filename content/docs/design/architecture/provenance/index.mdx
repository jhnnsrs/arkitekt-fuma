# Provenance & Audit Logging

The [Security](/docs/design/architecture/security) page makes a promise: that Arkitekt keeps a
sharp line between **you** and **the machine**, and that every change to your data is honestly
attributed. This page explains how that promise is kept.

The answer is **provenance-aware audit logging**. Every create, update, and delete on a tracked
record is captured as a history row, and each row is attributed to *who* made the change,
*which app* they used, and — crucially — the verified **task** it happened under. This lets an
administrator ask, after the fact, questions like:

> *"Which automated run touched this record, on whose authority, and what exactly did it change?"*

and get a precise, reliable answer.

<Callout type="info" title="Where this lives">
The provenance layer is implemented by a small Django library called **Koherent**, which the
Arkitekt services build on. If you are building or operating a backend service and want the
hands-on setup, see the [Koherent integration guide](/docs/developers/backend/koherent).
</Callout>

## What gets recorded

Every change produces a **provenance entry** that carries:

| Field               | Meaning                                                          |
|---------------------|------------------------------------------------------------------|
| `user`              | The human user the change is attributed to.                      |
| `client`            | The app / OAuth client that made the change, if any.             |
| `task`              | The verified task (assignation) the change ran under, if any.    |
| `kind`              | `CREATE`, `UPDATE`, or `DELETE`.                                 |
| `date`              | When the change happened.                                        |
| `effectiveChanges`  | The per-field old → new diff for that change.                    |

A change links to a `task`, and that task is where the real depth lives: it records not one
identity but several, how it relates to other tasks, and enough signed metadata to be verified
and reproduced later.

## How it fits together

Provenance is not bolted on at the end — it falls out of how the platform's layers stack. Each
layer has one job, and provenance is what you get when they are composed in order:

```
        Rekuest provenance token (signed, EdDSA)
                       │
        ┌──────────────▼───────────────┐
 kante  │  GraphQL transport + context  │   ← carries the request + its context
        └──────────────┬───────────────┘
                       │
        ┌──────────────▼───────────────┐
 authn. │  authentikate                 │   ← verifies the token, resolves the
        │  → user, client, organization │     acting user / client / organization
        └──────────────┬───────────────┘
                       │
        ┌──────────────▼───────────────┐
 koher. │  koherent                     │   ← stamps every change with user, client,
        │  → attributed history rows    │     and the verified task it ran under
        └───────────────────────────────┘
```

- **[kante](https://github.com/jhnnsrs/kante)** provides the GraphQL transport and the request
  context the upper layers build on.
- **[authentikate](https://github.com/jhnnsrs/authentikate)** owns identity. It verifies the
  signed provenance token and resolves the acting *user*, *client*, and *organization*. There is
  no bypass for provenance — the signature is always checked.
- **[koherent](https://github.com/jhnnsrs/koherent)** consumes that verified context and turns
  each data change into a durable, attributed history row, building a `Task` from the token's
  claims.

Because each layer depends on the one above it, the audit trail is only ever written from a
*verified* request — there is no path that records an unattributed or unsigned change.
